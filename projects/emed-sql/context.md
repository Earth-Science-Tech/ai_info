# emed_sql - SQL Schemas Context

## Overview
- **Purpose:** Canonical SQL schema definitions for the eMed platform — source of truth for two databases
- **Repo:** https://github.com/Earth-Science-Tech/emed_sql
- **Branch:** `main`

## Two Databases

| Snapshot Folder | Database | Role |
|-----------------|----------|------|
| `prod/` | `liberty_link_stage` | **Production** (despite the `_stage` name) |
| `dev/`  | `liberty_link_dev`   | Development / staging |

Both databases are on `liberty-link.database.windows.net`. `prod/` and `dev/` are auto-generated snapshots — one `.sql` file per object — produced by `python/extract_sql_files.py`. `migrations/` holds hand-written migration scripts applied dev → prod.

## File Organization

```
emed_sql/
├── prod/                            # auto-generated, mirrors liberty_link_stage
├── dev/                             # auto-generated, mirrors liberty_link_dev
├── migrations/                      # hand-written, applied dev → prod
├── create_emed_app_user.sql         # bootstrap (shared)
├── create_emed_etl_user.sql         # bootstrap (shared)
└── python/
    ├── extract_sql_files.py         # generator: --db prod|dev
    └── extract_schema.py            # markdown summary: --db prod|dev
```

Inside `prod/` and `dev/`, files follow flat prefix naming:

| Pattern | Purpose |
|---------|---------|
| `table_*.sql` | Table definitions (with mandatory fields, indexes inline) |
| `view_*.sql` | Views (CREATE OR ALTER) |
| `procedure_*.sql` | Stored procedures (CREATE OR ALTER) |
| `trigger_*.sql` | Triggers (CREATE OR ALTER) |
| `migration_grant_permissions_*.sql` | Per-object live-DB grants |
| `migration_fk_constraints.sql` | All FK constraints (single post-pass file) |
| `info.claude` | Markdown schema summary for AI consumption |
| `_GENERATED.md` | Snapshot timestamp + DB info |

## Generated File Banner

Each generated file starts with a stable banner — no timestamp, no DB name — so `diff prod/ dev/` surfaces only real schema differences:

```sql
-- =============================================================================
-- AUTO-GENERATED by emed_sql/python/extract_sql_files.py - DO NOT EDIT BY HAND
-- Object: TABLE dbo.moct_visit
-- =============================================================================
```

## Engineer Workflow (dev → prod)

1. Make schema change on `liberty_link_dev` (SSMS / sqlcmd).
2. Hand-write `migrations/<YYYY-MM-DD>_<description>.sql` capturing the change with idempotent guards.
3. `python python/extract_sql_files.py --db dev` to refresh `dev/`.
4. `diff dev/ prod/` — should reflect what the migration adds.
5. On `push prod`, the skill identifies the pending migration, runs it on prod, regenerates `prod/`, commits, and proceeds with the deploy.

## Critical Protocols

1. **All tables must include mandatory fields** (id, sql_user, date_created, date_modified, is_invalid)
2. **All new objects need GRANT statements** (see `skills/create-table.md`)
3. **Migrations are additive** — never destructive without a plan
4. **Never edit files in `prod/` or `dev/` by hand** — they're regenerated from the live DB
5. **All schema changes go through dev first** — direct edits to liberty_link_stage bypass the workflow
